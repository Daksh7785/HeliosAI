## Purpose of This Document

This document outlines how HeliosAI pushes live data to the Dashboard (`39_Dashboard.md`) without requiring the client to constantly poll the REST API (`32_API_Design.md`).

## Why WebSockets?

During a solar flare, flux values change rapidly. Constantly polling the REST API for new 1-second cadence data from SoLEXS and HEL1OS would overwhelm the database and introduce unnecessary latency. WebSockets provide a persistent, bidirectional connection for instantaneous telemetry and alert delivery.

## Architecture and Broker

The FastAPI backend maintains the WebSocket connections. However, because the backend is horizontally scaled across multiple pods (`51_Kubernetes.md`), a Pub/Sub broker is required to ensure a message generated by the Processing Subsystem reaches the specific FastAPI pod holding the user's WebSocket connection.
- **Broker:** Redis Pub/Sub.
- **Flow:** `services/intelligence/` writes to DB -> publishes to Redis -> `services/api/` consumes from Redis -> broadcasts to connected WebSockets.

## Channel Design

Clients subscribe to specific "topics":
- `channel:telemetry:fused`: Emits the latest synchronized light curve data points.
- `channel:alerts:nowcast`: Emits immediately when the Nowcasting Engine (`22_Nowcasting.md`) detects a flare.
- `channel:alerts:forecast`: Emits when the Forecasting Engine (`23_Forecasting.md`) updates probabilities.
